# 기능 명세서: 오늘의 할 일 웹 앱

**기능 브랜치**: `001-todo-app`

**작성일**: 2026-07-15

**상태**: Draft

**입력**: 사용자 설명: "오늘의 할 일" 웹 앱을 만들려고 해. 사용자 관점의 요구사항은 다음과 같아.

## 결정된 사항

- 데이터 영속성 방식은 브라우저 로컬 저장소를 사용하기로 결정했다. 구현 단계에서는 브라우저 저장소(localStorage 등)를 기준으로 한다.
- 서버 저장소는 이번 범위에 포함하지 않는다. 즉, 할 일 데이터는 단일 브라우저 환경에서 유지된다.

## 사용자 시나리오 및 테스트 (필수)

### 사용자 스토리 1 - 새 할 일을 생성하고 유효성 검증하기 (우선순위: P1)
사용자는 새 할 일을 입력하고 목록에 추가할 수 있다. 빈 값이나 공백만 있는 입력은 추가되지 않으며, 시스템은 사용자에게 안내 메시지를 보여준다.

**왜 이 우선순위인가**: 이 흐름은 앱의 핵심 가치이며, 사용자가 가장 먼저 기대하는 기본 동작이기 때문이다.

**독립 테스트**: 사용자는 앱에 진입한 뒤 새 할 일을 입력해 추가할 수 있고, 잘못된 입력에 대해 안내를 받는지 확인할 수 있다.

**수락 시나리오**:

1. **주어진 상황** 사용자가 할 일 목록이 비어 있는 상태일 때, **사용자가** 유효한 할 일을 입력하고 추가 버튼을 누르면, **그 결과** 새 할 일이 목록에 나타난다.
2. **주어진 상황** 사용자가 빈 문자열 또는 공백만 입력한 상태일 때, **사용자가** 추가 버튼을 누르면, **그 결과** 항목이 추가되지 않고 안내 메시지가 표시된다.

---

### 사용자 스토리 2 - 할 일을 완료/미완료 상태로 토글하고 삭제하기 (우선순위: P1)
사용자는 할 일을 완료 처리할 수 있고, 완료된 항목은 시각적으로 구분된다. 완료된 항목을 다시 미완료 상태로 되돌릴 수도 있다. 사용자는 삭제 전 확인 절차를 거쳐 할 일을 삭제할 수 있다.

**왜 이 우선순위인가**: 완료 상태 관리와 삭제는 할 일 앱의 핵심 상호작용이므로 가장 중요한 기능이다.

**독립 테스트**: 사용자는 한 항목을 완료 처리하고, 다시 되돌리며, 삭제 전 확인 메시지를 확인한 뒤 삭제가 수행되는지 검증할 수 있다.

**수락 시나리오**:

1. **주어진 상황** 사용자가 미완료 상태의 할 일이 있을 때, **사용자가** 해당 항목을 완료 처리하면, **그 결과** 항목이 완료 상태로 바뀌고 시각적으로 구분된다.
2. **주어진 상황** 완료 상태의 할 일이 있을 때, **사용자가** 해당 항목을 다시 선택하면, **그 결과** 항목이 미완료 상태로 되돌아간다.
3. **주어진 상황** 사용자가 삭제 버튼을 누른 상태일 때, **사용자가** 확인 절차를 완료하면, **그 결과** 항목이 목록에서 제거된다.

---

### 사용자 스토리 3 - 할 일 개수와 필터를 확인하고 빈 상태를 경험하기 (우선순위: P2)
사용자는 화면 상단에 남은 할 일 개수를 확인하고, 전체/진행중/완료 필터로 목록을 볼 수 있다. 할 일이 없을 때는 친절한 빈 상태 안내를 받는다.

**왜 이 우선순위인가**: 상태 요약과 필터링은 앱 사용성을 높이고, 빈 상태 안내는 비어 있는 목록에서의 혼란을 줄여준다.

**독립 테스트**: 사용자는 할 일을 추가/완료/삭제한 뒤 남은 개수와 필터 결과가 즉시 반영되는지 확인할 수 있다.

**수락 시나오**:

1. **주어진 상황** 할 일이 하나 이상 존재할 때, **사용자가** 항목을 추가 또는 변경하면, **그 결과** 상단의 남은 할 일 개수가 즉시 갱신된다.
2. **주어진 상황** 목록에 할 일이 없을 때, **사용자가** 화면을 확인하면, **그 결과** 빈 상태 안내 문구가 표시된다.
3. **주어진 상황** 사용자가 필터를 변경하면, **그 결과** 현재 선택된 필터에 맞는 항목만 표시된다.

---

### 사용자 스토리 4 - 데이터 지속성과 반응형 사용성 확보하기 (우선순위: P2)
사용자는 브라우저를 새로고침하거나 서버를 재시작해도 목록이 유지되고, 모바일 화면에서도 앱을 사용할 수 있다.

**왜 이 우선순위인가**: 데이터 지속성과 모바일 사용성은 실사용 만족도에 직접적인 영향을 주는 필수 품질 요소이기 때문이다.

**독립 테스트**: 사용자가 할 일을 추가한 뒤 새로고침하거나 앱을 재실행해도 목록이 유지되고, 모바일 폭에서 UI가 정상 동작하는지 확인할 수 있다.

**수락 시나오**:

1. **주어진 상황** 사용자가 할 일을 추가한 뒤 페이지를 새로고침하면, **그 결과** 목록이 유지된다.
2. **주어진 상황** 사용자가 360px 폭의 모바일 화면에서 앱을 사용하면, **그 결과** 주요 기능이 정상적으로 접근 가능하고 레이아웃이 깨지지 않는다.

---

### 엣지 케이스

- 빈 문자열, 공백만 있는 입력, 매우 긴 텍스트 입력이 들어올 때 시스템은 어떻게 동작하는가?
- 삭제 확인 절차를 취소했을 때 항목이 유지되는가?
- 모든 항목이 완료되었거나 모든 항목이 삭제되었을 때 남은 개수와 빈 상태 안내는 어떻게 표시되는가?
- 필터가 변경되었을 때 완료 상태와 미완료 상태의 항목이 올바르게 표시되는가?

## 요구사항 (필수)

### 기능 요구사항

- **FR-001**: 시스템은 사용자가 새 할 일을 입력해 목록에 추가할 수 있어야 한다.
- **FR-002**: 시스템은 빈 값 또는 공백만 있는 입력에 대해 추가를 거부하고, 사용자에게 안내 메시지를 표시해야 한다.
- **FR-003**: 시스템은 사용자가 할 일을 완료 처리할 수 있어야 하며, 완료된 항목은 시각적으로 구분되어야 한다.
- **FR-004**: 시스템은 완료된 항목을 다시 미완료 상태로 되돌릴 수 있어야 한다.
- **FR-005**: 시스템은 사용자가 할 일을 삭제할 수 있어야 하며, 삭제 전 확인 절차를 제공해야 한다.
- **FR-006**: 시스템은 화면 상단에 남은 할 일 개수를 항상 표시해야 하며, 추가/완료/삭제 동작 시 즉시 갱신되어야 한다.
- **FR-007**: 시스템은 전체, 진행중, 완료 3가지 필터를 제공해야 한다.
- **FR-008**: 시스템은 브라우저 새로고침 또는 브라우저 재시작 후에도 할 일 목록이 유지되도록 브라우저 로컬 저장소에 데이터를 저장해야 한다.
- **FR-009**: 시스템은 360px 폭의 모바일 화면에서도 사용 가능하도록 반응형 레이아웃을 제공해야 한다.
- **FR-010**: 시스템은 할 일이 하나도 없을 때 친절한 빈 상태 안내 문구를 표시해야 한다.

### 핵심 엔티티

- **할 일 항목(Todo Item)**: 사용자가 생성하고 관리하는 기본 단위로, 텍스트 내용, 완료 여부, 생성 시각을 포함한다.
- **필터 상태(Filter State)**: 현재 목록에 표시할 범위를 결정하는 상태로, 전체/진행중/완료 중 하나를 가진다.
- **알림 메시지(Notification Message)**: 입력 오류 또는 삭제 확인 등 사용자에게 제공되는 안내 메시지다.

## 성공 기준 (필수)

### 측정 가능한 결과

- **SC-001**: 사용자가 새 할 일을 1개 이상 추가하면 목록에 즉시 반영되고, 남은 할 일 개수가 올바르게 갱신된다.
- **SC-002**: 사용자가 할 일의 완료 상태를 토글하고 삭제할 때, 화면에 표시되는 상태와 개수가 일관되게 반영된다.
- **SC-003**: 사용자가 페이지를 새로고침하거나 서버를 재시작한 뒤에도 할 일 목록이 유지된다.
- **SC-004**: 360px 폭의 모바일 화면에서도 주요 기능에 접근하고, UI가 정상적으로 표시된다.
- **SC-005**: 할 일이 없을 때 빈 상태 안내가 표시되고, 사용자가 다음 행동을 이어갈 수 있다.

## 가정사항

- 사용자는 단일 사용자로 가정하며, 별도 로그인 또는 계정 관리 기능은 포함하지 않는다.
- 데이터 영속성은 브라우저 로컬 저장소를 사용한다. 서버 저장소는 이번 범위에 포함하지 않는다.
- 모바일 지원은 360px 폭을 기준으로 주요 동작이 가능하도록 구현한다.
- 기술 스택은 계획 단계에서 결정하며, 이 명세서의 범위는 사용자 요구사항 중심으로 유지한다.
