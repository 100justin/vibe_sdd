# Todo App Constitution

## Core Principles

### I. 스펙 우선
모든 기능은 구현 전에 스펙 문서로 정의한다. 스펙에 명시되지 않은 기능은 구현하지 않는다.

판정 기준:
- 새로운 기능을 시작하기 전에 관련 스펙 문서가 존재하고, 기능 범위가 명확히 기술되어 있는가?
- 구현 결과가 스펙에 정의된 범위를 벗어나지 않는가?

### II. 테스트 필수
핵심 기능(추가/완료 토글, 삭제, 남은 개수, 필터)은 pytest 자동화 테스트로 검증한다. 구현 완료의 기준은 해당 테스트가 통과하는 것이다.

판정 기준:
- 핵심 기능에 대한 pytest 테스트가 존재하는가?
- 테스트가 실제 동작을 검증하고 있으며, 실패 시 수정이 필요한가?
- 구현 완료 전 테스트가 통과하는가?

### III. 설정 분리
데이터베이스 경로 등 환경설정 값은 .env 파일로 관리하고 저장소에 커밋하지 않는다. 필요한 환경변수는 .env.example 파일에 문서화한다.

판정 기준:
- 민감하거나 환경별로 달라지는 값이 코드에 직접 하드코딩되어 있지 않은가?
- .env.example에 필요한 환경변수가 문서화되어 있는가?
- .env 파일이 저장소에 포함되지 않았는가?

### IV. 추적 가능한 커밋
모든 커밋 메시지 앞에 태스크 ID를 붙인다. 예: "T03: 할 일 생성 API 구현"

판정 기준:
- 모든 커밋 메시지가 태스크 ID로 시작하는가?
- 커밋 메시지가 변경 내용과 태스크를 함께 설명하는가?

### V. 단순성
요구사항에 없는 기능이나 라이브러리를 임의로 추가하지 않는다.

판정 기준:
- 추가된 기능이나 의존성이 현재 요구사항과 직접 관련이 있는가?
- 불필요한 라이브러리나 복잡한 구조가 도입되지 않았는가?

## 문서 작성 원칙
이 프로젝트에서 생성되는 모든 문서(constitution, spec, plan, tasks, checklist, 분석 리포트 등)는 반드시 한국어로 작성한다.

판정 기준:
- 생성된 문서의 본문이 한국어로 작성되어 있는가?
- 용어와 설명이 한국어로 일관되게 표현되는가?

## 개발 워크플로우
모든 변경은 스펙과 원칙에 따라 진행한다. 구현 전 테스트와 문서 기준을 확인하고, 변경 사항은 가능한 한 작은 단위로 나누어 관리한다.

판정 기준:
- 변경 전 스펙과 원칙을 확인했는가?
- 구현과 테스트가 함께 진행되는가?
- 문서가 한국어로 유지되고 있는가?

## Governance
이 Constitution은 프로젝트 개발 원칙의 최상위 기준으로 삼는다. 원칙을 위반하는 변경은 문서화된 이유와 함께 검토되어야 한다.
